Прочитай эти принципы перед началом работы и держи в голове во время каждой правки. Если предлагаешь что-то, что им противоречит — сначала останови себя и объясни юзеру почему стоит сделать исключение.

## 1. UX бота: минимум шума, максимум управляемости

- **Inline-кнопки вместо команд.** Слэш-команды только для базовых точек входа (`/start`, `/help`, `/cancel`). Любое действие глубже — кнопка в меню или в карточке.
- **Edit вместо новых сообщений.** Когда callback меняет состояние карточки/мастера/меню — делать `message.edit_text(...)`, не `message.answer(...)`. Если перейти на другой экран без редактирования невозможно — у предыдущего экрана убрать клавиатуру через `edit_reply_markup(None)`, чтобы юзер не тыкал старые кнопки и не получал дубли.
- **Одна «сарделька» статуса.** Прогресс по длинной операции (создание драфтов, scoring, bulk-book) идёт в одно сообщение через `progress_add()` из `src/bot/helpers.py` — не плодим простыню. Сообщения с inline-кнопками (drop-off picker, slot picker, обзорный экран) живут отдельно, но тоже редактируются на месте.
- **Anti-double-click.** Любое действие, которое запускает многошаговый процесс (создание поставки, синк каталога) должно ловить повторный клик. Стандартный паттерн в `ozon_book.py` — `_WIZARD_IN_FLIGHT` (ключ `request_id`, TTL).
- **Кнопка «◀ Назад» / «✖ Отмена»** обязательна в любом экране глубже двух уровней и в любом FSM-мастере.

## 2. Простота интерфейса важнее «красивых» фич

- Бот делается так, чтобы **любой селлер** мог понять его за минуту, не читая инструкции. Тексты на русском, конкретные, без жаргона. Если юзер увидит непонятный технический шум типа «warehouse_id #0» — это баг UX, переписывай.
- При сомнении «делать ли эту фичу детальнее» — обычно НЕ делать. Достаточная версия > правильная-но-сложная.
- Если кнопка может быть автоматизирована (бот сам выбирает разумный default) — автоматизируй и убери кнопку. Пример: «✓ Один drop-off для всех» как авто-выбор когда юзер уже выбрал хаб для предыдущего кластера.

## 3. Документация: WORKLOG + WHY, без лишнего

- **`WORKLOG.md` в корне репо** — журнал по сессиям. После каждого осмысленного блока правок дописывать запись формата `## YYYY-MM-DD HH:MM — Краткая суть` + что/почему/статус. Не затирать старое.
- **Память (`memory/`)** — для preferences пользователя, его профиля, текущего фокуса (live-state). Факты про код / коммиты / структуру — НЕ в память (есть `git log` и сам код). Код-конвенции — в `CLAUDE.md`.
- **Код-комментарии** только когда WHY не очевидно: скрытый инвариант, workaround, не-обвиозное число (60с между bulk-кластерами — потому что Ozon per-second лимит, обязательно объяснить). Что код делает — должны говорить имена идентификаторов, не комменты.
- Не плодить markdown-файлы (`PLAN.md`, `ANALYSIS.md`, `DECISIONS.md`) если юзер не попросил. Контекст — в `WORKLOG.md` + memory.

## 4. Коммиты — по команде юзера, не сами

- Локальные правки делаем свободно. **Коммитим только когда юзер явно сказал**: «коммить», «закоммить», «сделай commit».
- Перед коммитом проверить что нет в индексе секретов (`.env`, токены, ключи).
- Никогда не использовать `--no-verify`, `--no-gpg-sign`, force-push на main без явного «да форс на main».
- Пушим **мы сами** через SSH (см. [[git-setup-postavka-bot]]), не просим юзера выполнять `!git push`.

## 5. Без костылей, MVP-ready для масштабирования

- Бот сейчас однопользовательский, но проектируется так, чтобы переход на multi-tenant SaaS был лёгким:
  - Новые DB-таблицы: думать про `user_id`/`tenant_id` (можно nullable пока).
  - API-ключи и креды — через config (.env / БД), не в коде.
  - Бизнес-логика — agnostic к конкретному кабинету: интерпретируем Ozon/WB-ответы как «контекст текущего auth», без хардкодов брендов/категорий/регионов.
- **Магические числа и константы** (`warehouse_id`, `cluster_id`, имена складов) — никогда не зашивать в код. Тянуть из API или из БД. Если юзеру нужно настроить — давать ему UI/команду, не редактирование кода.
- **Лишнее стирать сразу.** Если переписал фичу — старый путь удалить, не оставлять как fallback и не помечать `# TODO: remove later`. Лучше пустая строка, чем мёртвый код.
- Если нашёл хак для частного кейса — задать вопрос «а как это будет работать для другого селлера / другого кабинета / другого региона». Чаще всего ответ показывает что нужна generic-реализация.

## 6. Автономность действий

- Правки в коде, перезапуски бота, чистка кэшей, редактирование `.env` — делать сразу, без подтверждения.
- Подтверждение спрашиваем **только для опасных и необратимых вещей**: `git push --force`, drop database, удаление файлов вне репо, отправка сообщений наружу (email, Slack, чужие API), смена paid-настроек.
- При архитектурном сомнении — предложить опции и выбрать дефолт самостоятельно, не блокироваться на AskUserQuestion по каждой мелочи.

## 7. Bootstrap для новой сессии

После рестарта Claude или новой сессии — читать в таком порядке:
1. Auto-memory (грузится сама) + `memory/MEMORY.md` индекс.
2. `memory/project_current_focus.md` — где остановились в прошлой сессии.
3. `WORKLOG.md` — 3-4 последние записи, подробности правок.
4. `git status` + `git log -5 --oneline` — состояние рабочей копии.
5. `data/screenshots/` — если юзер кидал картинку.

В конце каждой сессии **обновлять** `project_current_focus.md` (live-state, переписывается, не дописывается) и **дописывать** в `WORKLOG.md`.
